# Angular 21 Upgrade Plan

## Overview
Upgrade FOM from Angular v19.2.0 to Angular v21 across both **admin** and **public** frontends, including the shared **libs** package. This resolves security vulnerabilities and maintains compliance with LTS timelines (Angular 19 LTS ends May 19, 2026).

## Current State
- **Admin Frontend**: Angular ^19.2.0
- **Public Frontend**: Angular ^19.2.0
- **Shared Libs**: Angular ^19.0.0
- **Angular CLI/DevKit**: ^19.2.0
- **TypeScript**: ^5.8.3 (compatible with Angular 21)
- **Supporting packages**: Material ^19.2.0, CDK ^19.2.0, ng-bootstrap ^18.0.0, @angular/cdk ^19.2.0, etc.

## Upgrade Path: v19 → v21
This is a **2-major-version jump** requiring careful validation of dependencies and breaking changes.

### Key Considerations
1. **Breaking Changes**: Angular 19→20→21 includes syntax changes, deprecations, and API updates
2. **Dependency Compatibility**: Must verify all Angular-dependent packages support v21
3. **TypeScript**: v5.8.3 is supported by Angular 21 (no upgrade needed)
4. **Supporting Libraries**: Many need concurrent upgrades (Material, CDK, ng-bootstrap, @angular/localize, etc.)

## Known Angular Dependencies & Compatibility
Libraries requiring validation/upgrade for Angular 21 support:
- `@angular/material` - Currently ^19.2.0 (check v21 compatibility)
- `@angular/cdk` - Currently ^19.2.0 (Material peer dependency)

## Upgrade Plan

### Phase 1: Investigation & Planning
- [ ] Research Angular migration guides: https://angular.dev/update-guide (select v19→v21)
- [ ] Verify all direct dependencies support Angular 21
- [ ] Check npm/package repositories for Angular 21 compatible versions
- [ ] Review Angular 19→20→21 breaking changes documentation
- [ ] Identify potential code changes needed (signal adoption, new APIs, deprecations)

### Phase 2: Branch Setup
- [ ] Create feature branch: `feat/angular-21-upgrade`
- [ ] Ensure clean main baseline with latest dependencies from commit 4ed63d5c

### Phase 3: Dependency Upgrade (Sequential: libs → public → admin)
**Aggressive bumping strategy**: Upgrade all dependencies to latest compatible versions, including:
- Core Angular packages (v21)
- Material/CDK ecosystem
- Third-party libraries that support Angular 21
- Test/build tooling

For each package (libs, public, admin):
- [ ] **Update package.json** - Bump all @angular/* packages to latest v21.x
- [ ] **Aggressively upgrade supporting packages** to latest compatible versions:
  - `@angular-devkit/*` → ^21.0.0
  - `@angular/cli`, `@angular/compiler-cli` → ^21.0.0
  - `@angular-eslint/*` → ^21.0.0 (or latest if available)
  - `@angular/material`, `@angular/cdk` → ^21.0.0+ 
  - `@ng-bootstrap/ng-bootstrap` → latest v19+
  - `@ngx-bootstrap`, `@ngx-dropzone/*` → latest compatible
  - `jest-preset-angular` → latest compatible with Angular 21
  - `@fortawesome/angular-fontawesome`, `@auth0/angular-jwt`, `@rxweb/reactive-form-validators` → latest
- [ ] Run `npm install` to resolve transitive dependencies
- [ ] Check package-lock.json for validation

### Phase 4: Automatic Migrations (ng update)
For each app (public, admin):
- [ ] Run `ng update @angular/core --allow-dirty` to apply Angular schematics
- [ ] Run `ng update @angular/cli --allow-dirty` if needed
- [ ] Review and commit auto-migrated changes (template updates, deprecated APIs, etc.)

### Phase 5: Manual Code Fixes & Signal Migration
After migrations, address:
- [ ] **Build errors**: Resolve any TypeScript or compilation issues
- [ ] **Template changes**: Update deprecated Angular syntax
- [ ] **Deprecation warnings**: Fix deprecated API usages
- [ ] **Signal migration** (proactive modernization):
  - Identify component state candidates for signal conversion
  - Convert high-value components from RxJS observables to signals where appropriate
  - Update reactive forms usage to leverage new signal-based patterns
  - Migrate change detection strategy to `OnPush` where signals are adopted
  - Update component logic to use signal getters/setters instead of `@Input/@Output` where beneficial
- [ ] **Component adjustments**: Update component patterns to align with Angular 21 best practices

### Phase 5.5: Test Additions
Enhance test coverage during this upgrade:
- [ ] **Add tests for signal-based components**: Ensure signal migrations are properly tested
- [ ] **Add integration tests**: Cover critical user flows (authentication, data loading, navigation)
- [ ] **Add E2E tests** (if not already extensive): Test end-to-end workflows for admin and public frontends
- [ ] Review existing tests for outdated patterns and modernize as needed
- [ ] Target coverage improvement where gaps are identified

### Phase 6: Testing & Validation
- [ ] **Build verification**: Ensure `npm run build:admin` and `npm run build:public` succeed (production builds)
- [ ] **Unit tests**: Run `npm run test-unit` for each package (libs, admin, public); ensure all pass
- [ ] **New tests**: Run new tests added for signal components and integration paths
- [ ] **Manual testing**: Navigate all pages in both frontends to verify:
  - No console errors/warnings
  - UI rendering correct (look and feel intact)
  - Routing works properly
  - Forms and data binding functional
  - Authentication (Auth0/JWT) working
  - API communication functional
  - Leaflet maps and charts rendering
- [ ] **Performance check**: Compare bundle sizes pre/post upgrade (aggressive deps may increase size)

### Phase 7: Documentation & Deployment Prep
- [ ] Document all breaking changes, dependency upgrades, and signal migrations
- [ ] Add deployment notes covering:
  - **Build steps**: `npm ci && npm run build:admin && npm run build:public`
  - **Deployment environment requirements** (Node version, environment variables)
  - **Post-deployment verification steps** (smoke test URLs, health checks)
  - **Signal migration notes**: Document any components using new signal-based patterns
  - **Rollback procedures** if issues arise in production
- [ ] Verify CI/CD pipeline passes
- [ ] Create comprehensive PR with detailed summary:
  - List of all upgraded dependencies with version changes
  - Summary of signal migrations and affected components
  - Notable breaking changes and how they were resolved
  - Test additions and coverage improvements
  - Deployment notes (as above)
  - Closes #897

## PR Strategy: Single vs. Multiple PRs
Given the scope of this upgrade (aggressive dependency bumping + signal migration + test additions), consider:
- **Single PR approach** (if complexity is manageable):
  - Pros: Single review, atomic change set, cleaner history
  - Cons: Large diff, harder to review
- **Multi-PR approach** (if needed):
  - PR 1: Dependencies + automatic migrations (focus on compilation/build)
  - PR 2: Signal migrations (focus on code modernization)
  - PR 3: Tests and refinements
  
**Decision point**: After Phase 4, assess the diff size and complexity. If >50+ file changes with >5k LOC changes, consider splitting.
